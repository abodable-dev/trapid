# Feature Competitive Analysis & Restructure Proposal

## Current Issues with Feature Definitions

### 1. **Inconsistent Granularity**
- Some features are very broad ("Job Management", "Client Portal")
- Others are very specific ("Weather API", "GPS & Location Services")
- Competitors define features at different levels of detail

### 2. **Chapter Organization Problems**
- Current 12 chapters don't align with how competitors market their features
- Some chapters are too technical (e.g., "AI-POWERED INTELLIGENCE")
- Missing key categories that competitors emphasize

### 3. **Feature Comparison Challenges**
- Hard to compare "like with like" across competitors
- Some Trapid features don't exist in competitors (AI focus)
- Missing features that all competitors have (e.g., Pre-Construction Workflow)

## Key Changes Explained

### 1. **More Consistent Granularity**
- Each feature is now at a similar level of detail
- Features are defined as user-facing capabilities, not technical implementations
- Easier to compare with competitors

### 2. **Industry-Standard Organization**
- Chapters align with how competitors market features
- Follows the construction project lifecycle
- Easier for potential customers to understand

### 3. **Better Competitive Comparison**
- Features are defined at the same level across all platforms
- Can now accurately compare "Xero Integration" vs "Xero Integration"
- Missing features become immediately obvious

### 4. **Trapid Differentiators Highlighted**
- AI features grouped in Chapter 13 (Integrations & Automation)
- "Free Subcontractor Access" remains unique to Trapid
- Can easily identify where Trapid leads vs. follows

### 5. **Added Missing Categories**
- Pre-Construction Workflow (ClickHome has this, we don't)
- Tender Management (Wunderbuild has this)
- Warranty Management (BuilderTrend has this)
- Quality Inspections separate from Safety
- Payroll (DataBuild has full payroll)

- Add 91 new features across 13 chapters (7 features per chapter average)
- Set all `trapid_has` based on current implementation
- Mark competitor features based on research above

### Step 2: Migrate Existing Data
- Map old 61 features to new 91 features
- Preserve `dev_progress` percentages
- Keep historical data for reference

### Step 3: Update UI
- Update chapter names in frontend
- Ensure table handles 91 features gracefully
- Add chapter descriptions for context

### Step 4: Competitor Feature Mapping
- Research each competitor's feature pages
- Mark checkboxes for confirmed features
- Use "No information" for unclear features
